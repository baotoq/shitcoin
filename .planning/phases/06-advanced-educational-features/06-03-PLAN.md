    2. Add `demoDoubleSpend` method that runs the scripted scenario **in-process** (no subprocess spawning needed -- simpler and more reliable).

    **In-process double-spend demo flow:**
    The demo creates its own isolated environment using domain objects directly, avoiding the need for subprocesses. This is faster, more reliable, and shows the domain layer's protection directly.

    Steps:
    a. Print banner: "=== Double-Spend Attack Demo ==="
    b. Print explanation: "This demo shows how the blockchain prevents spending the same coins twice."
    c. Create a temp data directory using `os.MkdirTemp` for isolation.
    d. Create a new ServiceContext pointing to the temp directory (override storage paths in config).
    e. Create 3 wallets: Alice (miner), Bob, Charlie.
    f. Initialize chain with Alice as miner address.
    g. Mine 2-3 blocks so Alice has coins (using the chain directly, not subprocess).
    h. Print Alice's balance.
    i. Step 1: "Creating Transaction A: Alice sends 10 coins to Bob..."
       - Build TX-A using Alice's UTXOs, sending to Bob's address
       - Sign TX-A
       - Add TX-A to mempool -- prints SUCCESS
    j. Step 2: "Creating Transaction B: Alice tries to send the SAME coins to Charlie (double-spend!)..."
       - Build TX-B using the SAME UTXOs (same inputs as TX-A), sending to Charlie's address
       - Sign TX-B
       - Try to add TX-B to mempool -- prints the ERROR (ErrDoubleSpend)
       - Print: "REJECTED! The mempool detected the double-spend attempt."
    k. Step 3: "Mining a block to confirm Transaction A..."
       - Drain mempool and mine a block (pass totalFees from DrainByFee)
       - Show TX-A is confirmed in the block
    l. Step 4: "Trying Transaction B again after TX-A is confirmed..."
       - Try to add TX-B to mempool again
       - This time it fails with ErrUTXONotFound (the UTXOs were consumed by TX-A's block)
       - Print: "REJECTED! The UTXOs have been consumed by the confirmed transaction."
    m. Print educational summary:
       ```
       === Summary ===
       The blockchain prevents double-spending through two layers:
       1. MEMPOOL: Tracks which UTXOs are already claimed by pending transactions
       2. UTXO SET: Only unspent outputs can be used as inputs; once spent, they're gone

       This is why blockchain transactions are trustworthy without a central authority.
       ```
    n. Close the ServiceContext (bbolt DB) and clean up temp directory via `defer os.RemoveAll(tmpDir)`.

    **Update `cli.go`:**
    3. Add `case "demo"` to the `Run` switch statement, calling `c.demo(args[1:])`.
    4. Add demo to `printUsage()`: `"  demo doublespend           - Demonstrate double-spend detection"`

    **Important notes:**
    - Use domain objects directly (chain, mempool, wallet, utxo) -- no subprocesses needed
    - Use low difficulty (1 bit) for fast mining in demo
    - Override the config's InitialDifficulty to 1 for speed
    - Print each step with clear formatting and separators
    - Use `config.SatoshiPerCoin` for human-readable amounts (e.g., "10 coins" = 10 * SatoshiPerCoin satoshis)
    - The demo should complete in under 5 seconds
    - Close the ServiceContext and bbolt DB before removing temp dir
