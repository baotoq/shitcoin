# Phase 05.1: Upgrade to Go 1.26.1 - Research

**Researched:** 2026-03-06
**Domain:** Go version upgrade (1.24.0 -> 1.26.1)
**Confidence:** HIGH

## Summary

This phase upgrades the project from Go 1.24.0 to Go 1.26.1. The upgrade skips Go 1.25 entirely (two major versions). Go 1.26.1 is a patch release (2026-03-05) with security fixes on top of Go 1.26.0 (2026-02-10).

The project's codebase is low-risk for this upgrade. It does not use stdlib `crypto/ecdsa` directly (uses btcec/v2 instead), does not use `net/url.Parse`, does not use `image/jpeg`, and does not rely on any deprecated GODEBUG settings. The main opportunities are: (1) updating `go.mod` directive, (2) running `go fix` to apply modernizers, and (3) running `go mod tidy` to update dependencies. All direct dependencies (go-zero v1.10.0, bbolt v1.4.3, btcec/v2 v2.3.6, testify v1.11.1) are recent enough to work with Go 1.26.

**Primary recommendation:** Update go.mod to `go 1.26.1`, run `go mod tidy`, run `go fix ./...` for automatic code modernization, then run `go test ./...` to verify everything passes.

## Architecture Patterns

### Upgrade Procedure (Standard Go Practice)

```
1. Install Go 1.26.1 (or verify already installed)
2. Update go.mod: go 1.24.0 -> go 1.26.1
3. Run: go mod tidy
4. Run: go fix ./...
5. Run: go vet ./...
6. Run: go test ./...
7. Manual review of go fix changes
```

### What `go fix` Modernizers Will Change

Based on codebase analysis, these modernizers are relevant:

| Modernizer | What It Fixes | Files Affected |
|------------|--------------|----------------|
| `any` | `interface{}` -> `any` | `internal/infrastructure/persistence/jsonfile/wallet_repo_test.go` |
| `rangeint` | `for i := 0; i < N; i++` -> `for i := range N` | Multiple test files, `merkle.go`, `base58.go` |
| `minmax` | if/else min/max patterns -> `min()`/`max()` | Scan needed at fix time |

### Anti-Patterns to Avoid
- **Do not skip Go 1.25 review:** Even though we jump directly to 1.26.1, both Go 1.25 and 1.26 changes apply. The nil-pointer-before-error-check fix from Go 1.25 is relevant (code that deferences before checking error will now correctly panic).
- **Do not manually apply what `go fix` handles:** Let the tool do the work, then review.
- **Do not update toolchain directive separately:** The `go` directive in go.mod is sufficient; `toolchain` directive is optional and only needed for forward compatibility.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Code modernization | Manual find/replace of `interface{}`, old loop patterns | `go fix ./...` | Handles dozens of patterns safely, understands AST |
| Dependency compatibility check | Manual version checking | `go mod tidy` + `go test ./...` | Go module system handles minimum version selection |

## Common Pitfalls

### Pitfall 1: Go 1.25 Nil Pointer Behavior Change
**What goes wrong:** Code that dereferences a nil pointer before checking the error now correctly panics (was silently ignored in Go 1.21-1.24 due to compiler bug).
**Why it happens:** Compiler fix in Go 1.25 removes incorrect optimization that skipped nil dereferences.
**How to avoid:** Run `go test ./...` thoroughly. Any panic indicates a real bug that was hidden before.
**Warning signs:** Tests that pass on Go 1.24 but panic on 1.26.

### Pitfall 2: Green Tea GC Behavior Differences
**What goes wrong:** The new Green Tea garbage collector (default in 1.26) may change GC timing, affecting tests that depend on finalizer ordering or GC pressure.
**Why it happens:** New GC algorithm with 10-40% lower overhead runs collections differently.
**How to avoid:** This project has no finalizer-dependent code, so risk is negligible. If issues arise, `GOEXPERIMENT=nogreenteagc` is the escape hatch (removed in Go 1.27).
**Warning signs:** Intermittent test failures related to resource cleanup.

### Pitfall 3: Crypto rand Parameter Ignored
**What goes wrong:** In Go 1.26, `crypto/ecdsa.GenerateKey` and similar functions ignore the `rand` parameter.
**Why it happens:** Security hardening -- always uses crypto/rand internally.
**How to avoid:** This project uses `btcec.NewPrivateKey()` not stdlib crypto/ecdsa, so NOT AFFECTED. No action needed.

### Pitfall 4: Forgetting to Update go.sum
**What goes wrong:** Stale go.sum entries cause build failures or dependency confusion.
**How to avoid:** Always run `go mod tidy` after changing the go directive. This updates both go.mod require blocks and go.sum checksums.

### Pitfall 5: `go fix` Changing Semantics
**What goes wrong:** `go fix` transforms like `rangeint` could theoretically change semantics if the loop variable is modified within the body.
**How to avoid:** Review `go fix` diff before committing. The tool is conservative, but human review is warranted.

### Expected `go fix` Transformations

#### `interface{}` to `any` (wallet_repo_test.go)
```go
// Before
var parsed map[string]interface{}
arr, ok := wallets.([]interface{})
entry, ok := arr[0].(map[string]interface{})

// After (go fix any)
var parsed map[string]any
arr, ok := wallets.([]any)
entry, ok := arr[0].(map[string]any)
```

#### Range-over-int (various files)
```go
// Before
for i := 0; i < 5; i++ {
    // body using i
}

// After (go fix rangeint)
for i := range 5 {
    // body using i
}
```

Note: `go fix` will NOT transform loops where the loop variable is modified in the body, or where the upper bound changes during iteration (e.g., `len(level)` in merkle.go).
