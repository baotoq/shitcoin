<decisions>
## Implementation Decisions

### Assertion library
- Use `stretchr/testify` — assert + require + suite + mock packages
- No testify/suite framework globally — only where shared setup/teardown is useful (bbolt repos, P2P server tests)
- Adopt testify/mock to replace hand-rolled mock structs
- Keep hand-rolled mocks only where testify/mock adds unnecessary complexity

### Migration scope
- Migrate ALL 19 existing test files at once — no mixed styles
- Convert assertions AND refactor mocks to testify/mock
- Add testify/suite to bbolt repo tests and P2P tests (shared setup/teardown)
- Simple unit tests (e.g., difficulty_test.go, merkle_test.go) stay as plain functions with assert/require

### Assertion style
- `require` for setup/preconditions (e.g., require.NoError for file creation, DB init)
- `assert` for actual test assertions (e.g., assert.Equal for value comparisons)
- Assertion messages only when the assertion isn't self-explanatory (no message spam)
- Per-subtest `assert := assert.New(t)` inside each `t.Run` for table-driven tests

### Claude's Discretion
- Which specific mock structs to convert vs keep hand-rolled
- Exact suite grouping decisions for P2P tests
- Whether to consolidate duplicate test helpers

</decisions>
