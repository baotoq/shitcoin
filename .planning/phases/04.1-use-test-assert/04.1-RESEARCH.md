# Phase 04.1: Use Test Assert - Research

**Researched:** 2026-03-05
**Domain:** Go test migration (stdlib assertions to stretchr/testify)
**Confidence:** HIGH

## Summary

This phase migrates all 19 test files from raw `t.Errorf`/`t.Fatalf` stdlib assertions to `stretchr/testify` (assert, require, suite, mock packages). The codebase currently has ~3,000 lines of test code across 9 packages using manual comparison patterns like `if got != want { t.Errorf(...) }`.

The migration is mechanical and well-scoped: no production code changes, only test files. The main complexity lies in the P2P test files which have large hand-rolled mock structs (~170 lines for `fullMockChainRepo` + `mockUTXORepo` in relay_test.go) that are duplicated across chain_test.go and p2p test files. The `testify/suite` package fits naturally for bbolt repo tests (shared DB setup/teardown) and P2P server tests (shared server lifecycle).

**Primary recommendation:** Use `github.com/stretchr/testify` v1.10.x. Migrate in waves by package complexity: simple unit tests first, then infrastructure/domain tests with mocks, then P2P tests with suites.

## Architecture Patterns

### Pattern 1: Simple Unit Test Migration (assert/require only)

**What:** Replace `if got != want { t.Errorf(...) }` with `assert.Equal(t, want, got)`
**When to use:** Simple tests like merkle_test.go, difficulty_test.go, base58_test.go, pow_test.go
**Example:**
```go
// BEFORE
if root != (Hash{}) {
    t.Errorf("MerkleRoot of empty = %x; want zero hash", root)
}

// AFTER
assert.Equal(t, Hash{}, root)
```

### Pattern 2: Table-Driven Test Migration

**What:** Use `assert.New(t)` inside each subtest for cleaner table-driven tests
**When to use:** config_test.go, difficulty_test.go, base58_test.go, any tests using `t.Run` with `[]struct`
**Example:**
```go
for _, tt := range tests {
    t.Run(tt.name, func(t *testing.T) {
        assert := assert.New(t)
        require := require.New(t)

        got, err := SomeFunc(tt.input)
        require.NoError(err)
        assert.Equal(tt.expected, got)
    })
}
```

### Pattern 3: Suite for Shared Setup/Teardown (bbolt repos)

**What:** Use `suite.Suite` with `SetupTest` to create fresh DB per test
**When to use:** bbolt/chain_repo_test.go, bbolt/utxo_repo_test.go
**Example:**
```go
type ChainRepoSuite struct {
    suite.Suite
    db   *bolt.DB
    repo *BboltRepository
}

func (s *ChainRepoSuite) SetupTest() {
    dbPath := filepath.Join(s.T().TempDir(), "test.db")
    db, err := bolt.Open(dbPath, 0600, nil)
    s.Require().NoError(err)
    s.db = db
    s.T().Cleanup(func() { db.Close() })

    repo, err := NewBboltRepository(db)
    s.Require().NoError(err)
    s.repo = repo
}

func (s *ChainRepoSuite) TestSaveAndGetBlock() {
    ctx := context.Background()
    b := s.createTestBlock(block.Hash{}, 0, 8)

    s.Require().NoError(s.repo.SaveBlock(ctx, b))

    got, err := s.repo.GetBlock(ctx, b.Hash())
    s.Require().NoError(err)
    s.Assert().Equal(b.Hash(), got.Hash())
    s.Assert().Equal(b.Height(), got.Height())
}

func TestChainRepoSuite(t *testing.T) {
    suite.Run(t, new(ChainRepoSuite))
}
```

### Pattern 4: testify/mock for Repository Mocks

**What:** Replace hand-rolled mock structs with `mock.Mock`
**When to use:** Replace `fullMockChainRepo`, `mockUTXORepo`, `mockChainRepo`, `mockMempoolAdder`
**Example:**
```go
type MockChainRepo struct {
    mock.Mock
}

func (m *MockChainRepo) SaveBlock(ctx context.Context, b *block.Block) error {
    args := m.Called(ctx, b)
    return args.Error(0)
}

func (m *MockChainRepo) GetBlock(ctx context.Context, hash block.Hash) (*block.Block, error) {
    args := m.Called(ctx, hash)
    if args.Get(0) == nil {
        return nil, args.Error(1)
    }
    return args.Get(0).(*block.Block), args.Error(1)
}

// Usage in test:
mockRepo := new(MockChainRepo)
mockRepo.On("SaveBlock", mock.Anything, mock.AnythingOfType("*block.Block")).Return(nil)
mockRepo.On("GetBlock", mock.Anything, someHash).Return(someBlock, nil)
// ... test code ...
mockRepo.AssertExpectations(t)
```

### Mock Conversion Recommendations

| Current Mock | Location | Recommendation | Rationale |
|-------------|----------|----------------|-----------|
| `mockChainRepo` (server_test.go) | p2p | **Convert to testify/mock** | Simple stub, 9 methods, good mock.Mock fit |
| `fullMockChainRepo` (relay_test.go) | p2p | **Keep hand-rolled** | Has real in-memory storage logic (maps, mutex), not just return stubs |
| `reorgMockChainRepo` (reorg_test.go) | p2p | **Keep hand-rolled** | Extends fullMockChainRepo with undo storage, real logic |
| `mockUTXORepo` (relay_test.go) | p2p | **Keep hand-rolled** | Real in-memory UTXO store, used by integration-style tests |
| `mockChainRepo` (chain_test.go) | chain | **Keep hand-rolled** | Real in-memory store for chain reorg tests, needs actual state tracking |
| `mockUTXORepo` (chain_test.go) | chain | **Keep hand-rolled** | Real in-memory UTXO/undo store |
| `mockMempoolAdder` (chain_test.go) | chain | **Convert to testify/mock** | Simple capture mock, 1 method |

**Key insight:** The P2P and chain test mocks are _fake implementations_ (in-memory stores with real logic), not _stub mocks_ (return predefined values). testify/mock is ideal for stubs but adds boilerplate to fakes. Keep fakes hand-rolled but add assert/require to their test assertions. Convert only the simple stub mocks (server_test.go's mockChainRepo, mockMempoolAdder).

### Anti-Patterns to Avoid
- **Mixing assert and require randomly:** Use `require` only for preconditions/setup. Use `assert` for actual test assertions.
- **Adding assertion messages everywhere:** Only add messages when the assertion context is unclear. `assert.Equal(t, 5, height)` is self-documenting.
- **Using suite for simple tests:** Don't wrap `TestMerkleRoot_Single` in a suite -- plain functions with assert/require are cleaner.
- **Converting fakes to testify/mock:** Don't convert in-memory store mocks to testify/mock -- they have real logic that testify/mock can't express cleanly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Assertion helpers | Custom `assertEqual[T]`, `assertNoError` | `assert.Equal`, `require.NoError` | Testify provides better error messages, diff output |
| Error comparison | `if err != chain.ErrBlockNotFound` | `assert.ErrorIs(t, err, chain.ErrBlockNotFound)` | Handles wrapped errors correctly |
| Slice comparison | Manual loop comparing elements | `assert.Equal(t, expected, got)` for slices | Deep comparison with diff output |
| Nil checking | `if got == nil { t.Fatal(...) }` | `require.NotNil(t, got)` | Consistent, readable |

## Common Pitfalls

### Pitfall 1: assert vs require argument order
**What goes wrong:** `assert.Equal(t, got, want)` -- arguments reversed
**Why it happens:** Different convention from stdlib's `got, want` in error messages
**How to avoid:** Testify convention is `assert.Equal(t, expected, actual)` -- expected FIRST
**Warning signs:** Error messages showing "expected: [actual value], got: [expected value]"

### Pitfall 2: Forgetting to call suite.Run
**What goes wrong:** Suite test methods are never executed
**Why it happens:** Missing the `func TestXxxSuite(t *testing.T) { suite.Run(t, new(XxxSuite)) }` driver function
**How to avoid:** Every suite needs a Test function that calls `suite.Run`

### Pitfall 3: Using assert when require is needed
**What goes wrong:** Test continues executing after a fatal condition, causing nil pointer panics
**Why it happens:** Using `assert.NoError` for setup where `require.NoError` is needed
**How to avoid:** If the next line uses the result, use `require`

### Pitfall 4: mock.AnythingOfType with pointer types
**What goes wrong:** `mock.AnythingOfType("Block")` doesn't match `*block.Block`
**Why it happens:** Need to use the full type string including pointer
**How to avoid:** Use `mock.AnythingOfType("*block.Block")` or `mock.Anything` for simplicity

### Pitfall 5: Forgetting mock.AssertExpectations
**What goes wrong:** Mock expectations silently pass even when methods aren't called
**Why it happens:** Not calling `mockRepo.AssertExpectations(t)` at end of test
**How to avoid:** Always defer `mockRepo.AssertExpectations(t)` after setup

## Code Examples

### Common Assertion Conversions

```go
// Error checking
// BEFORE: if err != nil { t.Fatalf("failed: %v", err) }
// AFTER:
require.NoError(t, err)

// Value equality
// BEFORE: if got != want { t.Errorf("got %d; want %d", got, want) }
// AFTER:
assert.Equal(t, want, got)

// Boolean
// BEFORE: if !valid { t.Error("expected valid") }
// AFTER:
assert.True(t, valid)

// Nil check
// BEFORE: if result == nil { t.Fatal("expected non-nil") }
// AFTER:
require.NotNil(t, result)

// Error type check
// BEFORE: if err != chain.ErrBlockNotFound { t.Errorf("expected ErrBlockNotFound, got: %v", err) }
// AFTER:
assert.ErrorIs(t, err, chain.ErrBlockNotFound)

// Error expected
// BEFORE: if err == nil { t.Fatal("expected error, got nil") }
// AFTER:
require.Error(t, err)

// Length check
// BEFORE: if len(blocks) != 3 { t.Fatalf("got %d blocks, want 3", len(blocks)) }
// AFTER:
require.Len(t, blocks, 3)

// Zero value
// BEFORE: if root != (Hash{}) { t.Errorf(...) }
// AFTER:
assert.Equal(t, Hash{}, root)

// String contains (if needed)
// assert.Contains(t, errMsg, "expected substring")
```
