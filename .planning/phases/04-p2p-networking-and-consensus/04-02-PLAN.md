<task type="auto" tdd="true">
  <name>Task 1: Block and transaction relay handlers with validation and seen-hash tracking</name>
  <files>
    internal/domain/p2p/handler.go

  <behavior>
    - Receiving CmdTx with a valid signed transaction adds it to the local mempool and re-broadcasts inv to other peers
    - Receiving CmdTx with an invalid transaction (bad signature, missing UTXO) rejects it without relay
    - Receiving CmdBlock with valid PoW and correct prev-hash adds it to chain and re-broadcasts inv
    - Receiving CmdBlock with invalid PoW is rejected
    - Receiving CmdInv for an unknown block hash triggers CmdGetData request to the sender
    - Receiving CmdInv for an already-known hash does NOT trigger CmdGetData (seen tracking)
    - A block/tx hash seen from any peer is not requested again (prevents infinite loops)
  </behavior>
  <action>
    **server.go changes**: Add seen-hash tracking:
    - Add `seenBlocks map[string]bool` and `seenTxs map[string]bool` fields to Server (mutex-protected)
    - `func (s *Server) MarkSeen(typ string, hash string) bool` -- returns true if already seen, marks if not
    - Add `pow *block.ProofOfWork` field to Server (needed for block validation)
    - Add `func (s *Server) BroadcastBlock(blk *block.Block, excludeAddr string)` -- creates inv message with block hash, broadcasts to all peers except sender
    - Add `func (s *Server) BroadcastTx(transaction *tx.Transaction, excludeAddr string)` -- creates inv message with tx hash, broadcasts

    **handler.go changes**: Add relay message handlers:
    - `CmdInv` handler: parse InvPayload, for each hash check MarkSeen, if unseen send CmdGetData to requesting peer
    - `CmdGetData` handler: parse InvPayload, for "block" type look up block by hash from chainRepo and send CmdBlock response, for "tx" type look up from mempool and send CmdTx response
    - `CmdTx` handler: deserialize transaction from payload. The transaction needs to be transmitted as JSON. Validate by attempting `mempool.Add()` (which already validates signatures, UTXOs, double-spends). If accepted, broadcast inv to other peers (exclude sender).
    - `CmdBlock` handler: deserialize block from payload. Validate with `pow.Validate()`. Check prevHash matches our chain tip. If valid and extends our chain: apply UTXO changes via `utxoSet.ApplyBlock()`, save via `chainRepo.SaveBlockWithUTXOs()`, update chain tip, remove block's transactions from mempool, broadcast inv to other peers. If prevHash doesn't match tip, store for potential reorg handling (Plan 04 will implement this -- for now log a warning).

    **message.go changes**: Add helper for serializing/deserializing blocks and transactions for P2P:
    - Add `BlockPayload` struct that wraps block data in a JSON-serializable form (header fields + transactions). Use the same pattern as bbolt's BlockModel -- or simpler: serialize the block's header fields and raw transaction list.
    - Add `TxPayload` struct wrapping transaction data for network transmission.

    Note on block/tx serialization: The block and transaction types use unexported fields. Create P2P-specific payload structs that mirror the domain types with exported JSON fields, plus `ToBlock()`/`FromBlock()` conversion methods. Follow the same pattern as `infrastructure/persistence/bbolt/models.go`.

    **p2p_test.go additions**:
    - `TestTxBroadcast`: 3 nodes connected in a line (A-B-C). Create and send a valid tx on A. Verify it appears in C's mempool.
    - `TestBlockBroadcast`: 2 connected nodes. Mine a block on node A. Verify node B's chain height increases.
    - `TestValidation`: Send invalid block (wrong PoW) to a node. Verify it's rejected (chain height unchanged).
    - `TestSeenTracking`: Send same inv twice. Verify only one getdata request.

<task type="auto">
  <name>Task 2: Wire send and mine CLI commands to broadcast via P2P server</name>
  <files>
    internal/handler/cli/cli.go
    internal/domain/p2p/server.go
  </files>
  <action>
    **cli.go changes**: The `startNode` method (updated in Plan 01) creates a P2P Server. Now wire the server into the mining and transaction flows:

    - Store the P2P server reference in a way accessible to the auto-mine loop. The simplest approach: pass the server to a new `startNodeWithP2P` flow that:
      1. After mining a block in the auto-mine loop, calls `server.BroadcastBlock(blk, "")` to announce to all peers
      2. The `send` CLI command currently adds to mempool only. In `startnode` mode, after `mempool.Add()`, also call `server.BroadcastTx(transaction, "")` to announce to peers.

    - Since `send` is a separate CLI invocation (not inside startnode), the broadcast happens on the receiving side: when a transaction is added to the mempool via P2P (CmdTx handler), it's already re-broadcast by the handler (Task 1). The user runs `send` on the node locally, which adds to mempool. The server's periodic or event-driven mechanism is not needed here -- the `send` command is run on the same node, so the transaction is already in the local mempool. Broadcasting happens when the user runs send on a node that has an active P2P server.

    - Approach: Add a `Server` field to the CLI struct (nil when not in startnode mode). In `startNode`, set `c.server = server`. In `send`, after `mempool.Add()`, if `c.server != nil`, call `c.server.BroadcastTx(transaction, "")`.

    - In the auto-mine loop inside `startNode`, after `chain.MineBlock()`, call `server.BroadcastBlock(blk, "")`.

    - Add `server.AddBlockHandler(func(blk *block.Block))` callback to Server so that when a block is received from a peer while mining, the node can cancel the current mining attempt and accept the peer's block. This prevents the race condition in Pitfall #3 from the research. Use a mutex on chain state mutations.

    **server.go changes**: Add `onBlockReceived` callback field:
    - `func (s *Server) OnBlockReceived(fn func(*block.Block))` -- registers callback
    - In CmdBlock handler, after successfully applying the block, call the callback
    - The CLI's auto-mine loop uses this callback to cancel its current mining context
