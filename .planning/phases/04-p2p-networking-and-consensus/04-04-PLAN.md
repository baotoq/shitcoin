<objective>
Implement fork detection and chain reorganization with UTXO reversal.

Purpose: This is the consensus mechanism -- when two nodes mine competing blocks, the network converges on the longest valid chain. This completes the distributed blockchain by handling the most complex scenario: undoing work and reapplying a different chain fork.
Output: When a node detects a peer has a longer valid chain, it undoes its shorter fork, applies the longer one, and correctly updates all UTXO state.
</objective>

  <behavior>
    - Reorganize from height 5 (tip A5) to a 6-block fork (fork at height 3, new blocks B4-B6): undoes A5, A4, applies B4, B5, B6
    - After reorg, chain tip is B6 and height is 6
    - UTXOs from A4-A5 are restored (undo), UTXOs from B4-B6 are applied
    - Transactions from orphaned blocks A4-A5 that are not in B4-B6 are returned for mempool re-add
    - GetUndoEntry retrieves the undo entry for a specific block height
    - DeleteBlocksAbove(height) removes all blocks above the given height from storage
  </behavior>
  <action>
    **repository.go changes**: Extend the Repository interface with two new methods:
    ```go
    // GetUndoEntry retrieves the UTXO undo entry for a block at the given height.
    GetUndoEntry(ctx context.Context, blockHeight uint64) (*utxo.UndoEntry, error)

    // DeleteBlocksAbove removes all blocks above the given height.
    // Used during reorganization to remove orphaned blocks.
    DeleteBlocksAbove(ctx context.Context, height uint64) error
    ```

    **bbolt/repository.go changes**: Implement the two new methods:
    - `GetUndoEntry`: Read from the undo bucket using the height key (same format as SaveUndoEntry)
    - `DeleteBlocksAbove`: Iterate the height index from height+1 upward. For each block: delete from blocks bucket (by hash), delete height index entry, delete undo entry. Use a single bbolt Update transaction for atomicity.

    **chain.go changes**: Add the `Reorganize` method:
    ```go
    func (c *Chain) Reorganize(ctx context.Context, forkHeight uint64, newBlocks []*block.Block, mempool MempoolAdder) error
    ```
    Where `MempoolAdder` is a minimal interface `{ Add(*tx.Transaction) error }` to avoid hard coupling.

    The method does:
    1. **Find orphaned blocks**: Get blocks from `forkHeight + 1` to current tip using `GetBlocksInRange`
    2. **Collect orphaned transactions**: Extract all non-coinbase transactions from orphaned blocks
    3. **Undo orphaned blocks** (reverse order, from tip down to forkHeight+1):
       - For each height: `GetUndoEntry(height)` then `utxoSet.UndoBlock(entry)`
    4. **Delete orphaned blocks** from storage: `DeleteBlocksAbove(forkHeight)`
    5. **Apply new blocks** (forward order):
       - For each new block: validate PoW, apply UTXO changes, save with undo entry
    6. **Update chain tip** to the last new block
    7. **Re-add orphaned transactions**: For each orphaned tx not present in new chain's transactions, attempt `mempool.Add()` (ignore errors -- some may now be invalid)
    8. Return nil on success

    The entire operation should be logged with `slog` for debugging (which blocks undone, which applied).

    **chain_test.go**: Unit test Reorganize using in-memory repositories:
    - Build a chain of 5 blocks
    - Create a fork at height 3 with 3 new blocks (heights 4, 5, 6)
    - Call Reorganize
    - Verify tip is the new block at height 6
    - Verify UTXO state reflects the new chain (orphaned block UTXOs undone, new block UTXOs applied)

  <action>
    **handler.go changes**: Update the CmdBlock handler (from Plan 02) to handle fork scenarios:
    - When a received block's prevHash does NOT match our chain tip:
      1. Check if we know the block's prevHash (look up in chainRepo by hash)
      2. If unknown, this block may be part of a longer fork -- request the peer's chain backwards from this block
      3. If known at some height, we have a fork point. The peer's chain extends from that fork point.
      4. Request remaining blocks from the peer: send `CmdGetBlocks{StartHeight: forkHeight + 1, EndHeight: 0}` to get the full fork
      5. Once all fork blocks are received, compare total chain length:
         - If peer's chain (forkHeight + number of new blocks) > our height, trigger `chain.Reorganize(ctx, forkHeight, newBlocks, mempool)`
         - If our chain is longer or equal, discard the fork blocks (we win)
    - After successful reorg, broadcast the new tip to other peers

    **server.go changes**:
    - Add `forkBlocks map[string][]*block.Block` to buffer incoming fork blocks during reorg detection (keyed by peer addr)
    - Add `func (s *Server) findForkHeight(blockHash block.Hash) (uint64, error)` -- walks back from the given hash to find where it diverges from our chain. Start from the received block's prevHash, check if it's in our chain. If not, request the previous block from peer. Repeat until we find a common ancestor.
    - An alternative simpler approach: When we receive a block that doesn't extend our tip, ask the peer for their chain height. If higher, request blocks from our height backward using a new `CmdGetHeaders`-style approach, OR simply request all blocks from height 1 and find the fork point by comparing hashes. For simplicity in this educational project: request blocks from the peer starting at `ourHeight - 10` (or 0 if height < 10) to find the divergence point, then reorganize.

    Simpler practical approach for educational project:
    - When receiving a block that doesn't connect to our tip, store it in a "orphan block" buffer
    - Request the peer's full chain info: send GetBlocks with StartHeight=0, compare block hashes to find fork point
    - This is inefficient at scale but clear and educational
    - Once fork point found and peer chain is longer, collect all blocks from fork point+1 in correct order, call Reorganize
